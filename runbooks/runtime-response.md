# Runtime Response Runbook

## Default Flow

The default production-like response is alert-first:

```text
Falco -> Falcosidekick -> webhook -> GitHub Issue -> human triage
```

Auto-quarantine is intentionally not enabled by default. A false positive should not isolate production workloads without approval.

## Demo: Generate A Falco Alert

Run a shell inside a voting pod:

```powershell
kubectl -n voting exec deploy/vote -- sh -c "id"
```

Expected result:

- Falco matches `Shell Spawned In Voting Namespace`.
- Falcosidekick forwards the alert to the configured webhook.
- A GitHub issue or incident notification is created.
- Alert timestamp is captured for the report.

## Demo: Open Incident Manually

Use this when the webhook is not configured during a cost-saving local demo:

```powershell
gh workflow run runtime-incident.yml `
  -f severity=warning `
  -f title="Falco shell detected in voting pod" `
  -f namespace=voting `
  -f pod=vote-demo `
  -f details="Demo alert generated by kubectl exec"
```

## Optional Quarantine Mode

Only enable quarantine for a controlled demo. The action must be scoped to the `voting` namespace.

Minimum manual containment command:

```powershell
kubectl -n voting label pod <pod-name> security.devsecops/quarantine=true --overwrite
kubectl -n voting apply -f response/k8s/quarantine-networkpolicy.yaml
```

Do not target `kube-system`, `argocd`, `monitoring`, `logging`, `falco`, or `gatekeeper-system`.

## Evidence To Capture

- Falco event timestamp.
- GitHub issue URL or alert destination screenshot.
- Loki query result for the affected pod.
- Grafana SLI/SLO panel during the incident.
- Decision log: alert-only or quarantine.
